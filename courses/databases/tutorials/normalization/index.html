Introduction to databases — Tutorial 2

Normalization

In this tutorial you’ll learn:

  • How to obtain a non-redundant set of functional dependencies.
  • How to determine the candidate keys of a table given its functional dependencies.
  • How to determine the normal form of a table.

Prerequisites:

  • Having attended Lecture 2.

1 Non-redundant functional dependencies

Consider the following set \(\mathcal{F}\) of functional dependencies:

  • \(A, B \rightarrow C\)

  • \(D \rightarrow B, C\)

  • \(A \rightarrow B\)

Exercise

Exercise 1.1 Derive a minimal set \(\mathcal{G}\) of functional dependencies that is equivalent to \(\mathcal{F}\).

Solution

Based on the definition of a set of minimal functional dependencies (slide 18 in Lecture 2), we can rewrite the functional dependencies so that the right side consists of only one attribute. \(D \rightarrow B, C\) is equivalent to \(D \rightarrow B\) and \(D \rightarrow C\).

  • \(A, B \rightarrow C\)

  • \(D \rightarrow B\)

  • \(D \rightarrow C\)

  • \(A \rightarrow B\)

All functional dependencies except the first one is trivially left-irreducible (the determinant consists of only one column). The first has two attributes in the determinant, therefore we need to check whether we can eliminate one of the two columns and still preserve an equivalent set of functional dependencies.

We try to apply Armstrong’s axioms to compute the closure of this set of functional dependencies.

We have \(A \rightarrow B\). By augmentation we obtain:

\[A \rightarrow A, B\]

We have \(A, B \rightarrow C\). By transitivity we obtain:

\[A \rightarrow C\]

Therefore, the column \(B\) in \(A, B \rightarrow C\) is not useful and we can drop it.

The set \(\mathcal{G}\) consists of the following FDs:

  • \(A \rightarrow C\)

  • \(D \rightarrow B\)

  • \(D \rightarrow C\)

  • \(A \rightarrow B\)

2 Candidate keys and normal forms (1)

We consider the following table:

Patient (ssn, first_name, last_name, phone_number, insurance_number, 
         insurance_expiration_date)

where the following set \(\mathcal{F}\) of functional dependencies holds:

\[ \begin{align} ssn \rightarrow first\_name, & last\_name, phone\_number, insurance\_number, \\ & insurance\_expiration\_date \end{align} \]

\[ insurance\_number \rightarrow insurance\_expiration\_date \]

Exercise

Exercise 2.1 Derive a minimal set \(\mathcal{G}\) of functional dependencies that is equivalent to \(\mathcal{F}\).

Solution

First, we need to rewrite the FDs in canonical form.

  1. \(ssn \rightarrow first\_name\)
  2. \(ssn \rightarrow last\_name\)
  3. \(ssn \rightarrow phone\_number\)
  4. \(ssn \rightarrow insurance\_number\)
  5. \(ssn \rightarrow insurance\_expiration\_date\)
  6. \(insurance\_number \rightarrow insurance\_expiration\_date\)

The determinant of each FD is composed of only one column, therefore it is already irreducible. It is easy to see that all FDs with ssn as a determinant must be kept (otherwise we lose some information).

By transitivity from 4. and 6. we obtain:

\[ssn \rightarrow insurance\_expiration\_date\]

Therefore, \(\mathcal{G}\) is obtained from \(\mathcal{F}\) by removing 5.

Exercise

Exercise 2.2 Given \(\mathcal{G}\), identify the candidate keys in the table Patient.

Solution

From the functional dependencies in \(\mathcal{F}\), it’s easy to see that that the only column that implies all the others is ssn. Therefore, {ssn} is the only candidate key in this table.

Exercise

Exercise 2.3 Specify the normal form of the table Patient. Justify your answer.

Solution

  • It is immediate to verify that the table is 1NF.

  • The table is 2NF because there is only one candidate key, which is composed of only one column.

  • The table is not in 3NF. Indeed, there is a functional dependency between two non-prime columns:

\[insurance\_number \rightarrow insurance\_expiration\_date\]

Exercise

Exercise 2.4 How would you obtain two or more tables in BCNF from the table Patient?

Solution

We need to split the data relative to the patient from the data relative to the insurance. Therefore, we propose the following two tables:

Patient (ssn, first_name, last_name, phone_number, insurance_number)
Insurance (insurance_number, insurance_expiration_date)

Note that the column insurance_number in table Patient is foreign key to the column insurance_number in table Insurance.

3 Candidate keys and normal forms (2)

Let \(R\) be a relational table with five columns \((A, B, C, D, E)\). The following set \(\mathcal{F}\) of functional dependencies hold:

  1. \(A, B \rightarrow C\)
  2. \(C \rightarrow A\)
  3. \(C \rightarrow B\)
  4. \(C \rightarrow D\)
  5. \(D \rightarrow E\)

Exercise

Exercise 3.1 Specify the candidate keys of the table \(R\).

Solution

First, let’s try sets composed of only one column: \(\{A\}\), \(\{B\}\), \(\{C\}\), \(\{D\}\) and \(\{E\}\).

We have the following: (\(\{X\}^+_{\mathcal{F}}\) indicates the set of all columns implied by \(X\)).

  • \(\{A\}^+_{\mathcal{F}} = \{A\}\)

  • \(\{B\}^+_{\mathcal{F}} = \{B\}\)

  • \(\{C\}^+_{\mathcal{F}} = \{A, B, C, D, E\}\)

  • \(\{D\}^+_{\mathcal{F}} = \{D, E\}\)

  • \(\{E\}^+_{\mathcal{F}} = \{E\}\)

Therefore, \(\{C\}\) is a candidate key because it implies all the other columns.

From the functional dependency 1., we obtain that \(\{A, B\}\) implies \(C\); therefore, by transitivity they imply all the other columns.

In conclusion, we have two candidate keys: \(\{C\}\) and \(\{A, B\}\).

Exercise

Exercise 3.2 We assume that \(R\) is in 1NF.

  • Is table \(R\) in 2NF? Justify your answer.

  • Is table \(R\) in 3NF? Justify your answer.

Solution

  • \(R\) is in 2NF. In fact, all non-prime columns depend entirely on both candidate keys.

  • \(R\) is not in 3NF. In fact, the functional dependency \(D \rightarrow E\) is between two non-prime columns.

4 Candidate keys and normal forms (3)

Let \(R\) be the a relational table with five columns \((A, B, C, D, E)\). The following set \(\mathcal{F}\) of functional dependencies hold:

  • \(A \rightarrow C\)
  • \(A, B \rightarrow D\)
  • \(A, B \rightarrow E\)

Exercise

Exercise 4.1 Specify the candidate keys of the table \(R\).

Solution

The following hold:

  • \(\{A\}^+_{\mathcal{F}} = \{A, C\}\)

  • \(\{B\}^+_{\mathcal{F}} = \{B\}\)

  • \(\{C\}^+_{\mathcal{F}} = \{C\}\)

  • \(\{D\}^+_{\mathcal{F}} = \{D\}\)

  • \(\{E\}^+_{\mathcal{F}} = \{E\}\)

It is clear that the only possible candidate key could be {A, B}; let’s verify it by computing the closure of {A, B} under \(\mathcal{F}\):

  • \(\{A, B\}^+_{\mathcal{F}} = \{A, B, C, D, E\}\)

Indeed, {A, B} is the candidate key.

Exercise

Exercise 4.2 We assume that \(R\) is in 1NF.

  • Is table \(R\) in 2NF? Justify your answer.

  • Is table \(R\) in 3NF? Justify your answer.

Solution

  • \(R\) is not in 2NF. In fact, the non-prime column \(C\) is functionally dependent on only one part of the candidate key.

  • \(R\) is not in 3NF, because it doesn’t fulfill the first condition, that is being in 2NF.
