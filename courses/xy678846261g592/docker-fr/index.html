Introduction à Docker

Dans ce TD, vous apprendrez les bases de Docker :

  • exécuter des conteneurs.
  • créer des images.
  • créer et utiliser des volumes.
  • créer et utiliser des réseaux.

Prérequis :

  • Avoir installé Docker sur votre ordinateur ou avoir importé la machine virtuelle Linux soit avec VirtualBox, soit avec Multipass.
Vous utilisez une machine virtuelle avec Multipass ? Cliquez ici pour plus d’infos

Commandes Multipass

  • Pour démarrer la machine virtuelle, saisissez la commande multipass start cloudvm.

  • Le dossier /home/ubuntu/labs dans la machine virtuelle est monté (c’est-à-dire lié) au dossier sur VOTRE ordinateur (appelons-le LAB) où vous stockerez tous les fichiers que vous créérez pendant les TD et les TP. Vous avez spécifié ce dossier lors de l’installation de la machine virtuelle.

  • Si vous ne vous souvenez plus du chemin vers le dossier lié à /home/ubuntu/labs, saisissez la commande multipass info cloudvm.

  • Pour ouvrir un terminal dans la machine virtuelle, saisissez la commande multipass shell cloudvm.

  • Lorsque le terminal s’ouvre, le dossier courant est /home/ubuntu.

  • Saisissez la commande cd labs pour vous déplacer vers le dossier où vous trouverez votre matériel de TP.

  • Vous utiliserez le terminal de la machine virtuelle uniquement pour saisir les commandes Docker. Vous pouvez utiliser tous les outils installés sur votre machine pour créer et gérer les fichiers dans le dossier LAB directement depuis votre ordinateur.

  • Une fois le TP terminé, saisissez la commande exit pour quitter le terminal de la machine virtuelle. Vous reviendrez alors au terminal de votre ordinateur.

  • Dans le terminal de votre ordinateur, saisissez la commande multipass stop cloudvm pour arrêter la machine virtuelle. Cela ne détruira PAS vos fichiers ! Cela arrêtera simplement la machine virtuelle pour qu’elle n’utilise plus inutilement les ressources de votre ordinateur.

Terminologie

  • Vous utiliserez le terminal pour exécuter des commandes Docker. Le terminal est le client qui communique avec le démon Docker.

  • Docker exécute des conteneurs sur votre ordinateur. Nous ferons référence à votre ordinateur comme étant l’hôte, les conteneurs étant les invités.

  • Une application conteneurisée est une application qui s’exécute dans un conteneur.

1 Exécuter des conteneurs

Nous allons apprendre à utiliser la commande docker run ainsi que quelques-unes de ses options.
Dans les exercices suivants, nous exécuterons des conteneurs à partir de l’image nommée alpine qui est disponible sur le registre DockerHub.
Cette image fournit une distribution légère (c’est-à-dire qu’elle ne contient pas beaucoup de fonctionnalités) de Linux.

Exercice

Exercise 1.1 Vous souhaitez exécuter un conteneur à partir de la dernière version de l’image alpine.
Quelle commande devez-vous saisir dans le terminal ?
Exécutez-la.
Solution

Le but de cet exercice est de commencer à jouer avec la commande docker run. Comme la question ne dit rien à propos des options, et ne mentionne pas non plus la commande à exécuter à l’intérieur du conteneur, on saisirait :

docker run alpine

Exercice

Exercise 1.2 Le conteneur est-il toujours en cours d’exécution ?
Solution

Afin d’afficher tous les conteneurs en cours d’exécution sur l’hôte, saisissez la commande suivante :

docker container ls

Votre conteneur ne devrait pas apparaître dans le résultat, car il n’est pas en cours d’exécution. Pour voir tous les conteneurs, y compris ceux qui ne sont pas en cours d’exécution, saisissez la commande suivante :

docker container ls -a

Exercice

Exercise 1.3 Lors du lancement du conteneur, quelle commande a été exécutée ? Pouvez-vous dire pourquoi le conteneur s’est arrêté sans donner aucune sortie ?
Solution

La commande est /bin/sh ; le conteneur exécute un terminal Linux. Mais comme nous n’avons pas spécifié ce qu’il fallait faire avec ce terminal (nous n’avons lancé aucune commande Linux, ni essayé d’accéder au terminal), le conteneur s’est arrêté.

Nous allons maintenant faire quelque chose d’utile avec l’image alpine. Assurez-vous de lire les bonnes pratiques que vous devriez adopter lorsque vous utilisez des images et des conteneurs.

Bonnes pratiques

  1. Nommez vos conteneurs. Bien que Docker attribue un nom par défaut à chaque conteneur, il est généralement recommandé de donner à un conteneur un nom de votre choix pour le rendre facilement distinguable. Vous pouvez le faire en utilisant l’option --name. Essayez ce qui suit :

docker run --name my-alpine alpine

Comme précédemment, le conteneur s’arrête immédiatement. Si vous affichez tous vos conteneurs en saissant la commande :

docker container ls -a

vous devriez voir un conteneur nommé my-alpine.

  1. Supprimez automatiquement un conteneur si vous allez l’utiliser une seule fois. À moins que vous ne vouliez réutiliser votre conteneur plus tard, vous pouvez demander à Docker de le supprimer automatiquement lorsqu’il s’arrête en utilisant l’option --rm. Cela empêchera les conteneurs inutilisés de prendre trop d’espace sur disque.

Essayez ce qui suit :

docker run --rm --name container-to-remove alpine

Si vous affichez tous les conteneurs, vous devriez voir qu’il n’y a pas de conteneur nommé container-to-remove.

  1. Supprimez les conteneurs que vous n’utilisez plus. Les conteneurs arrêtés qui ont été exécutés sans l’option --rm sont toujours stockés sur votre machine hôte. Si vous voulez supprimer un conteneur spécifique (par exemple, my-alpine), utilisez la commande suivante :

docker container rm my-alpine

Si vous souhaitez supprimer tous les conteneurs arrêtés, utilisez la commande suivante :

docker container prune

  1. Supprimez les images inutilisées. Les images peuvent prendre beaucoup d’espace sur disque. Par conséquent, vous devriez penser à supprimer celles que vous n’avez plus l’intention d’utiliser. Les commandes pour supprimer une image spécifique et élaguer les images inutilisées sont respectivement docker image rm et docker image prune -a.

1.1 Passer une commande à une application conteneurisée

Rappelez-vous que la commande docker run suit le modèle suivant :

docker run [options] nom-image [commande] [args]

Le paramètre optionnel commande fait référence à une commande que vous pouvez passer à l’application conteneurisée, éventuellement avec quelques arguments (paramètre args).

Voyons un exemple. Comme nous l’avons vu précédemment, lorsque nous lançons un conteneur à partir de l’image alpine, un terminal Linux /bin/sh est lancé.

Remarque

Le terminal Linux /bin/sh est exécuté dans le conteneur. Par la suite, nous utiliserons les termes suivants :

  • Terminal hôte Le terminal que vous utilisez pour interagir avec le système d’exploitation de votre ordinateur.
  • Terminal invité. Le terminal qui s’exécute dans le conteneur.

En utilisant le paramètre optionnel commande, nous pouvons exécuter une commande dans le terminal invité.

Exercice

Exercise 1.4 Lancez un conteneur à partir de l’image alpine et exécutez la commande Linux ls (cette commande affiche les fichiers stockés dans un dossier). Où sont stockés les fichiers affichés ? Dans l’hôte ou dans le conteneur ?

Solution

docker run --rm --name ls-test alpine ls

La commande ls est exécutée dans le terminal invité, par conséquent les fichiers affichés sont stockés dans le conteneur.

Remarque

Dans l’exercice 1.4 la commande ls est exécutée dans le terminal invité, mais sa sortie est redirigée vers le terminal hôte.

En d’autres termes, lorsque nous lançons le conteneur, nous n’interagissons pas directement avec le terminal invité ; nous envoyons juste une commande et la sortie est redirigée vers le terminal hôte.

On va maintenant voir comment exécuter une commande avec arguments dans le terminal invité.

Exercice

Exercise 1.5 Exécutez un conteneur à partir de l’image alpine et vérifiez via la commande Linux ping si le site Web www.centralesupelec.fr est accessible.

Solution

docker run --rm --name ping-test alpine ping www.centralesupelec.fr

Pour interrompre ping , il suffit de saisir la combinaison de touches que vous utilisez pour interrompre n’importe quelle autre commande dans votre terminal (typiquement Ctrl-C sous Windows et Cmd-C sous macOs).

Maintenant, exécutez la commande suivante :

docker run --name my-alpine --rm -it alpine

Exercice

Exercise 1.6 Qu’obtenez-vous ?
Solution

Lorsque nous exécutons un conteneur à partir de l’image alpine, la commande /bin/sh est exécutée dans le conteneur. Comme nous avons spécifié l’option -it, ce que nous obtenons est un accès au terminal Linux fonctionnant dans le conteneur.

2 Création d’images

Une image de conteneur est constituée d’un système de fichiers en couches, chacune étant immuable ; cela signifie que les fichiers qui sont stockés dans une couche ne peuvent pas être modifiés ou supprimés, et que des fichiers ne peuvent pas être ajoutés à une couche.

Lorsqu’un conteneur est créé à partir d’une image, il sera composé de toutes les couches en lecture seule de l’image et, par-dessus celles-ci, d’une couche en écriture (appelée couche conteneur), où tous les nouveaux fichiers créés dans le conteneur seront écrits.

2.1 Comprendre un Dockerfile

Un Dockerfile est un fichier texte qui contient toutes les instructions nécessaires pour créer une image. C’est grâce à un Dockerfile que la création d’images est une tâche automatisée et répétable.

Le Dockerfile suivant permet de créer une image basée sur Nginx (un serveur Web), et de personnaliser celle-ci pour servir un site web statique avec une configuration dédiée.

FROM nginx:alpine
RUN rm -rf /usr/share/nginx/html/*
COPY index.html /usr/share/nginx/html/
COPY nginx.conf /etc/nginx/conf.d/default.conf
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]

Chaque ligne correspond à une instruction distincte ; les commandes en majuscules sont celles que Docker peut interpréter et exécuter lors de la construction de l’image.

Voici une explication de chaque ligne :

  • FROM nginx:alpine. On utilise une image légère de Nginx basée sur Alpine Linux comme point de départ.

  • RUN rm -rf /usr/share/nginx/html/*. Supprime le contenu HTML par défaut fourni avec Nginx afin de repartir d’un dossier vide. Cette commande est bien exécutée dans l’image et pas dans le système de fichier de la machine hôte.

  • COPY index.html /usr/share/nginx/html/. Ajoute la page web personnalisée dans le répertoire servi par Nginx.

  • COPY nginx.conf /etc/nginx/conf.d/default.conf. Remplace la configuration Nginx par défaut avec une configuration personnalisée.

  • EXPOSE 80. Indique que le conteneur écoute sur le port 80 (HTTP).

  • CMD ["nginx", "-g", "daemon off;"]. Lance Nginx en mode premier plan pour que le conteneur reste actif.

Exercise 2.1

Téléchargez cet archive et décompressez-la.

Quelle commande devez-vous exécuter pour créer une image nommée mynginx à partir de ce Dockerfile?

Solution

Exercice

Exercise 2.2 Saisissez la commande docker history mynginx. Qu’affiche cette commande?

Solution

Cette commande affiche les instructions ayant servi à construire l’image. On peut notamment y reconnaître celles du Dockerfile que nous venons de créer, ainsi que remonter à l’historique des instructions utilisées pour générer l’image de base.

Il convient d’ignorer le terme missing affiché à côté de certaines instructions : il s’agit d’un comportement hérité des premières versions de Docker, qui n’est plus pertinent aujourd’hui.

Exercice

Exercise 2.3 Exécutez la commande docker inspect mynginx. Combien de couches (layers) obtenez-vous ? Quelle relation observez-vous entre le nombre de lignes du Dockerfile et le nombre de couches ? Enfin, quelle relation pouvez-vous établir entre le nombre de couches et le nombre d’instructions affichées par la commande docker history ?

Solution

Nous comptons 11 couches, ce qui est supérieur au nombre de lignes du Dockerfile. Cela s’explique par le fait que l’image inclut également les couches de l’image de base.

On observe aussi que le nombre d’instructions affichées par la commande docker history est nettement supérieur au nombre de couches. En général, seules les instructions de build (comme RUN, COPY ou WORKDIR) donnent lieu à la création d’une couche, tandis que certaines instructions, comme EXPOSE, ne correspondent à aucune couche.

On pourrait aussi le vérifier en créant l’image sans l’instruction EXPOSE.

On peut exécuter un conteneur à partir de l’image mynginx via la commande suivante :

docker run -p 8080:80 mynginx

L’option -p définit un port mapping : le port 80 exposé par l’interface réseau du conteneur est redirigé vers le port 8080 de la machine hôte. Cela permet aux applications de la machine hôte de communiquer avec le conteneur. Par exemple, vous pouvez afficher le contenu du fichier index.html en ouvrant un navigateur web et en accédant à l’URL localhost:8080.

Exercice

Exercise 2.4 Arrêtez le conteneur en utilisant la combinaison de touches Ctrl+C (Windows) ou Cmd+C (macOS). Exécutez ensuite la commande suivante :

docker run -p 8080:80 mynginx ls.

Qu’obtenez-vous ? Pourquoi ?

Solution

Une liste de fichiers s’affiche. La commande ls est exécutée à la place de celle définie par l’instruction CMD du Dockerfile.

Cela n’a rien d’étonnant. En effet, l’instruction CMD sert à définir des arguments par défaut pour le conteneur. Si un argument est fourni lors de l’exécution du conteneur (dans ce cas, la commande ls), il remplace complètement celui spécifié dans CMD.

Exercice

Exercise 2.5

  • Modifiez le Dockerfile pour remplacer CMD par ENTRYPOINT.

  • Recréez l’image mynginx.

  • Exécutez la commande docker run -p 8080:80 mynginx ls.

Qu’obtenez-vous maintenant ? Pourquoi ?

Solution

On obtient une erreur. En réalité, la commande ls est interprétée comme un argument de la commande nginx définie par l’instruction ENTRYPOINT, et Nginx ne reconnaît pas cette option.

On constate ainsi que le fait de passer ls en argument ne remplace pas la commande définie par ENTRYPOINT.

Cet exemple illustre la différence entre ENTRYPOINT et CMD : le premier sert à définir la commande principale exécutée lors du lancement d’un conteneur, tandis que le second permet de définir des arguments par défaut, qui peuvent être remplacés par ceux fournis par l’utilisateur.

2.2 Conteneuriser une application Python

Téléchargez cette archive et décompressez-la dans votre dossier de travail.

Dans cette archive, vous trouverez :

  • Un Dockerfile.
  • Un script Python main.py qui demande à l’utilisateur d’entrer l’URL et la langue d’une page Web, et affiche les 10 mots les plus fréquents dans cette page.
  • Un fichier requirements.txt contenant la liste des modules Python nécessaires pour exécuter le script donné.

Le contenu du Dockerfile est le suivant :

FROM python:3.7-slim
RUN mkdir -p /app
WORKDIR /app
COPY ./main.py ./requirements.txt /app/
RUN pip install -r requirements.txt
ENTRYPOINT ["python", "main.py"]

Exercice

Exercise 2.6 Décrivez ce que fait ce Dockerfile.
Solution
  • Prend python:3.7-slim comme image de base.
  • Crée un nouveau dossier app dans l’image sous le dossier racine.
  • Modifie le dossier de travail en /app.
  • Copie les fichiers main.py et requirements.txt vers le dossier /app stocké dans l’image.
  • Exécute la commande pip install pour installer les bibliothèques Python spécifiées dans le fichier requirements.txt.
  • Exécute la commande python main.py.

Exercice

Exercise 2.7 Créez une image wordfreq en utilisant ce Dockerfile.
Solution

docker build -t wordfreq .

Exercice

Exercise 2.8 Sans modifier le Dockerfile, récréez la même image. Que constatez-vous ?
Solution

La création est très rapide. Comme nous n’avons pas changé le Dockerfile, l’image est récréée en utilisant les couches de l’image créée précédemment. Ceci est clairement indiqué par le mot CACHED écrit à côté de chaque couche. L’utilisation des couches déjà stockées est appelée build cache.

Exercice

Exercise 2.9 Que se passe-t-il si vous modifiez une ligne du script Python et que vous récréez l’image ?
Solution

Ajoutez une instruction à la fin de main.py, comme par exemple :

print("Finish!")

Ensuite récréez l’image. Les trois couches inférieures ne sont pas affectées par la modification, donc elles bénéficient du build cache. La couche 4 est la première à être affectée par la modification. Cette couche, ainsi que les couches supérieures, doivent donc être récrées.

Exercice

Exercise 2.10 Sur la base des considérations précédentes, pouvez-vous dire ce qui ne va pas avec ce Dockerfile ? Modifiez le Dockerfile en conséquence et recréez l’image.
Solution

À chaque modification de main.py suivie d’une reconstruction de l’image, les couches 4 et 5 sont recréées, ce qui entraîne le téléchargement et l’installation de l’ensemble des modules Python. Selon leur nombre et leur taille, cela peut prendre du temps.

Une meilleure approche consiste à installer les dépendances avant de copier le script Python dans l’image. Voici comment adapter le Dockerfile :

FROM python:3.7-slim
RUN mkdir -p /app
WORKDIR /app
COPY ./requirements.txt /app/
RUN pip install -r requirements.txt
COPY ./main.py /app/
ENTRYPOINT ["python", "main.py"]

Exercice

Exercise 2.11 Modifiez main.py en ajoutant une nouvelle ligne de code et récréez l’image. Qu’est-ce qui a changé ?
Solution

Les paquets Python ne sont pas réinstallés ; la création de l’image prend beaucoup moins de temps qu’auparavant.

Jouez avec l’application en exécutant la commande suivante :

docker run --rm -it wordfreq

L’application conteneurisée vous invite à saisir l’URL d’une page web ainsi que sa langue (en anglais). Elle affiche ensuite les 20 mots les plus fréquemment utilisés sur cette page.

3 Volumes

Les données créées dans un conteneur sont éphémères : elles sont stockées dans la couche conteneur et sont supprimées lorsque le conteneur est supprimé.

Bon à savoir

Si vous arrêtez un conteneur sans le supprimer, les données sont conservées. Elles ne sont supprimées que lorsque le conteneur est effacé.

Afin de rendre le cycle de vie des données indépendant de celui des conteneurs, il est possible d’utiliser des volumes.

Un volume peut être considéré comme un périphérique de stockage virtuel attaché à un conteneur. Tous les fichiers sont écrits sur un volume qui ne sera pas supprimé lorsque le conteneur est supprimé.

Créons un nouveau volume Docker appelé data-volume :

docker volume create data-volume

Bon à savoir (notion avancée)

Où les données seront-elles réellement stockées ?

Vous pouvez inspecter le nouveau volume en saisissant la commande suivante :

docker volume inspect data-volume

Un point de montage est indiqué ; il s’agit du dossier dans lequel les données seront effectivement stockées. Si vous êtes sou Linux, ce dossier sera disponible sur l’hôte ; si vous êtes sous Windows ou MacOS, vous ne trouverez pas ce dossier sur votre ordinateur. Il sera disponible dans la machine virtuelle que Docker Desktop utilise sur MacOS et Windows.

Voulez-vous voir le dossier ? (Instructions pour Windows et MacOS)

Une façon de voir la VM cachée est d’exécuter l’application conteneurisée suivante :

docker run -it --rm --privileged --pid=host justincormack/nsenter1

Cette application ouvrira un terminal invité dans la VM. Vous pouvez ensuite utiliser les commandes cd et ls pour naviguer jusqu’au dossier indiqué comme chemin de montage du nouveau volume.

Un volume Docker peut être utilisée pour permettre à deux conteneurs Docker de partager des données.

4 Réseaux

Afin de permettre aux conteneurs de communiquer et, par conséquent, de coopérer, Docker définit un modèle de réseau simple connu sous le nom de container network model (voir la figure).

Si vous utilisez une machine virtuelle Linux avec Multipass

Dans cette section, vous devrez ouvrir plusieurs terminaux dans la machine virtuelle. Vous pouvez le faire facilement en utilisant byobu, un gestionnaire de fenêtres avancé déjà disponible dans votre machine virtuelle.

  • saisissez simplement byobu pour lancer le gestionnaire de fenêtres.

  • Si vous voulez ouvrir un nouveau terminal, appuyez simplement sur F2.

  • Si vous voulez passer d’un terminal à un autre, appuyez simplement sur F3 (pour passer au terminal précédent) ou F4 (pour passer au terminal suivant).

  • Si vous voulez fermer un terminal, saisissez simplement exit.

  • Lorsque vous fermez tous les terminaux, byobu cesse de s’exécuter.

Figure 4.1: Le modèle de réseau de Docker.

Exercice

Exercise 4.1 Décrivez la sortie de la commande suivante :

docker network ls

Solution

La commande répertorie tous les réseaux créés par Docker sur votre ordinateur. Pour chaque réseau, les valeurs de quatre attributs sont affichées :

  • L’identifiant.
  • Le nom.
  • Le pilote utilisé par le réseau.
  • La portée du réseau (locale ou globale). Une portée locale signifie que le réseau relie des conteneurs fonctionnant sur le même hôte.

Il est possible que vous voyez différents réseaux, si vous avez déjà utilisé Docker avant ce cours. Cependant, trois réseaux méritent d’être signalés :

  • Le réseau nommé bridge, qui utilise le pilote bridge et a une portée locale. Par défaut, tout nouveau conteneur est attaché à ce réseau.
  • Le réseau nommé host, qui utilise le pilote host et a une portée locale. Il est utilisé lorsque nous voulons qu’un conteneur utilise directement l’interface réseau de l’hôte. Il est important de se rappeler que ce réseau ne doit être utilisé que pour analyser le trafic réseau de l’hôte. Dans les autres cas, l’utilisation de ce réseau expose le conteneur à toutes sortes de risques de sécurité.
  • Le réseau nommé none, qui utilise le pilote null et a une portée locale. Attacher un conteneur à ce réseau signifie que le conteneur n’est connecté à aucun réseau, et qu’il est donc complètement isolé.

Exercice

Exercise 4.2 La commande suivante :

docker network inspect bridge

affiche la configuration du réseau bridge. En regardant cette configuration, pouvez-vous dire quelles adresses IP seront données aux conteneurs attachés à ce réseau ? Quelle est l’adresse IP du routeur de ce réseau ?
Solution

Les informations sont spécifiées dans le champ nommé IPAM, plus précisément :

Sous-réseau indique la plage d’adresses IP utilisée par le réseau. La valeur de ce champ doit être 172.17.0.0/16 ; les adresses vont de 172.17.0.1 à 172.17.255.255.

Gateway indique l’adresse IP du routeur du réseau. La valeur doit être 172.17.0.1.

4.1 Création de réseaux

Par défaut, tout nouveau conteneur est rattaché au réseau nommé bridge.

Exercise 4.3 Expliquez pourquoi ce n’est pas une bonne pratique d’attacher tous nos conteneurs au même réseau.

Solution

Tous les nouveaux conteneurs pourront communiquer sur ce réseau. Ce n’est pas une bonne idée. Si un attaquant parvient à compromettre l’un de ces conteneurs, il pourrait être en mesure d’attaquer également les autres conteneurs. En règle générale, nous devrions attacher deux conteneurs au même réseau uniquement s’il est nécessaire qu’ils communiquent.

Nous pouvons utiliser la commande suivante pour créer un réseau.

docker network create network_name

Exercice

Exercise 4.4 Créez deux réseaux nommés buckingham et rochefort qui utilisent le pilote bridge (voir figure 4.1). En utilisant la commande docker network inspect, regardez les adresses IP des nouveaux réseaux et notez-les.
Solution

Exécutez les commandes suivantes :

docker network create buckingham

docker network create rochefort

Les adresses IP du réseau buckingham sont 172.18.0.0/16 (adresses de 172.18.0.1 à 172.18.255.255) ; Les adresses IP du réseau rochefort sont : 172.19.0.0/16 (en supposant que vous créez buckingham avant rochefort).

Les adresses IP peuvent être différentes sur vos machines.

Exercice

Exercise 4.5 Créez trois conteneurs athos, porthos et aramis et attachez-les aux deux réseaux buckingham et rochefort comme indiqué dans la figure 4.1. Les trois conteneurs ouvriront un shell Linux Alpine. Vous devrez lancer les commandes dans trois onglets distincts de votre terminal.

  • Quelles seront les adresses IP des trois conteneurs dans les deux réseaux ? Rappelez-vous que porthos est attaché à deux réseaux, il aura donc deux interfaces réseau (endpoints) et, par conséquent, deux adresses IP.
  • Vérifiez vos réponses en inspectant les deux réseaux (utilisez la commande docker network inspect).
Solution

Voici les commandes pour exécuter athos et aramis tout en les connectant à buckingham et rochefort respectivement.

docker run --rm -it --name athos --network buckingham alpine

docker run --rm -it --name aramis --network rochefort alpine

Voici la commande pour exécuter porthos et l’attacher à buckingham :

docker run --rm -it --name porthos --network buckingham alpine

La commande suivante attache porthos au second réseau rochefort :

docker network connect rochefort porthos

En ce qui concerne les adresses IP, chaque réseau a des adresses IP dans l’intervalle 172.x.0.0/16, où x est 18 dans le réseau buckingham et 19 dans le réseau rochefort. L’adresse 172.x.0.1 est réservée au routeur. Par conséquent, les conteneurs se verront attribuer des adresses IP à partir de 172.x.0.2. Dans cette solution, nous avons créé athos, aramis et portos dans cet ordre. Par conséquent, les adresses IP seront les suivantes :

  • Réseau buckingham:
    • athos: 172.18.0.2
    • porthos: 172.18.0.3
  • Réseau rochefort:
    • aramis: 172.19.0.2
    • porthos: 172.19.0.3

Vous pouvez vérifier cette configuration en inspectant les deux réseaux avec les commandes suivantes :

docker network inspect buckingham

docker network inspect rochefort

Les adresses IP peuvent être différentes sur vos machines.

Les seuls conteneurs qui ne peuvent pas communiquer sont athos et aramis, parce qu’ils ne sont pas connectés au même réseau.

Exercice

Exercise 4.7 Essayez d’envoyer un ping à porthos depuis athos en utilisant son adresse IP.

  • Quelle adresse IP de porthos utiliseriez-vous ?
Solution

Nous devons utiliser l’adresse IP attribuée à l’endoint reliant porthos au réseau buckingham, auquel athos est connecté. Dans notre cas, il s’agit de 172.18.0.3.

Exercice

Exercise 4.8 Essayez d’envoyer un ping à porthos depuis athos en utilisant son nom. Réussissez-vous ? Êtes-vous surpris ?
Solution

Nous y parvenons. En effet, le réseau buckingham fournit un serveur DNS, qui peut traduire les noms en adresses IP.

Vous pouvez maintenant arrêter les conteneurs.
