1 Docker Compose

Dans cette section, vous allez créer et déployer une application multiservice en utilisant Docker Compose.

Téléchargez cette archive et décompressez-la dans un dossier de votre ordinateur. L’archive contient tous les fichiers nécessaires pour créer et exécuter une application web composée de deux services :

  • web. C’est le frontend (la partie avec laquelle l’utilisateur interagit) de l’application. Il se compose de code HTML et JavaScript.

  • db. C’est le backend (la partie cachée à l’utilisateur). Il s’agit d’une base de données (relationnelle) PostgreSQL.

La structure de l’application est présentée dans la figure 1.1. Le dossier racine de l’application contient deux sous-dossiers, un pour chaque service (database et web).

Figure 1.1: Structure de l’application.

Bon à savoir

  • Les fichiers avec l’extension .conf dans le dossier database contiennent les paramètres de configuration de la base de données PostgreSQL.

  • Le fichier init-db.sql dans le dossier database contient les requêtes SQL pour alimenter la base de données avec des données (photos de chats).

  • Le dossier web contient le code HTML et JavaScript de l’application web. Le fichier package.json contient les dépendances à installer.

Les deux dossiers incluent un Dockerfile chacun. Le Dockerfile qui se trouve dans le dossier database est le suivant :

FROM postgres
COPY init-db.sql /docker-entrypoint-initdb.d/
RUN chown postgres:postgres /docker-entrypoint-initdb.d/*.sql
ENV POSTGRES_USER dockeruser
ENV POSTGRES_PASSWORD dockerpass
ENV POSTGRES_DB pets

Ce Dockerfile utilise comme base une image existante appelée postgres, qui est documentée ici.

Exercice

Exercise 1.1 Considérez l’instruction suivante :

COPY init-db.sql /docker-entrypoint-initdb.d/

En lisant la documentation de l’image postgres, répondez aux deux questions suivantes :

  1. Où se trouve le dossier /docker-entrypoint-initdb.d/ ?

  2. Pourquoi copions-nous le fichier init-db.sql vers ce dossier ?

Solution

  • Le dossier existe déjà dans l’image de base.

  • En regardant la documentation, nous apprenons que nous pouvons placer n’importe quel script d’initialisation dans ce dossier. En d’autres termes, au démarrage, la base de données exécutera les requêtes SQL contenues dans le fichier init-db.sql. afin que la base soit alimentée en nouvelles données.

La documentation nous apprend également que le script n’est pas exécuté si le dossier de données n’est pas vide. Cela signifie que les bases de données existantes ne seront pas touchées.

Les trois dernières lignes du fichier Docker contiennent un mot-clé (ENV) que nous n’avons jamais rencontré auparavant.

Exercice

Exercise 1.2 Lisez à nouveau la documentation de l’image postgres et essayez d’expliquer la signification des trois dernières lignes du Dockerfile.

Solution

Ces lignes définissent la valeur de trois variables d’environnement, utiles pour passer certains paramètres au gestionnaire de base de données (SGBD). La documentation nous apprend que :

  • POSTGRES_USER est le nom d’utilisateur du SGBD. S’il n’est pas défini, l’utilisateur par défaut est postgres.

  • POSTGRES_PASSWORD est le mot de passe associé à l’utilisateur. C’est la seule variable obligatoire.

  • POSTGRES_DB est le nom donné à la base de données. S’il n’est pas spécifié, le nom de la base de données sera le même que le nom d’utilisateur.

Docker secrets

La spécification de données sensibles (par exemple, des mots de passe et des clés API) via des variables d’environnement définies dans des Dockerfiles est largement considérée comme une mauvaise pratique de sécurité.

En effet, les valeurs de ces variables d’environnement sont stockées dans les couches de l’image, ce qui les rend accessibles à toute personne disposant d’un accès à cette image.

Une approche plus sécurisée consiste à utiliser les Docker secrets. Les secrets sont des objets Docker dédiés, conçus pour stocker des informations sensibles séparément des images de conteneurs. Une fois créés, l’accès à un secret peut être accordé à un service spécifique. À l’exécution, les instances du service reçoivent le secret via un système de fichiers en mémoire, évitant ainsi tout stockage persistant au sein du conteneur.

Il convient de noter que, lors de l’utilisation de Docker Compose, les secrets ne sont pas chiffrés au repos. Pour bénéficier du chiffrement, il est nécessaire d’utiliser Docker Swarm. Vous en apprendrez davantage sur l’utilisation des secrets dans le cadre du TP.

Le Dockerfile de l’application Web est le suivant :

FROM node:9.6-alpine
RUN mkdir /app
WORKDIR /app
COPY package.json /app/
RUN npm install
COPY ./src /app/src
EXPOSE 3000
CMD node src/server.js

Le Dockerfile utilise l’image node:9.6-alpine comme base ; Cette image contient un environnement Node.js, permettant l’exécution de JavaScript côté serveur. Les instructions du Dockerfile ressemblent à celles des exemples que nous avons vus dans le premier TD.

Bon à savoir

  • La commande npm install installe toutes les dépendances spécifiées dans le fichier package.json. Les bibliothèques sont téléchargées depuis le registre du gestionnaire de paquet npm.

  • L’instruction EXPOSE 3000 informe que le conteneur sera disponible au port 3000 lorsqu’il sera exécuté à partir de l’image.

Dans la documentation de Docker nous apprenons que l’instruction EXPOSE ne publie pas le port. Elle a vocation à être une sorte de documentation à destination des utilisateurs de l’image.

Afin de publier réellement le port, nous utiliserons le fichier docker-compose.yaml.

1.1 Description de l’application

L’application que nous considérons présente les éléments constitutifs d’une animalerie en ligne. Dans la version actuelle, l’application se limite à montrer quelques photos de chats.

Le dossier racine de l’application comprend un fichier nommé docker-compose.yaml qui contient la configuration de l’application. Le contenu du fichier est le suivant. Il s’agit d’une séquence de paires clé-valeur. Chaque clé est aussi appelé un champ.

services:
  web:
    build: web
    image: pet-store-web
    networks:
      - backend
    ports:
      - 8080:3000
  db:
    build: database
    image: pet-store-db
    networks:
      - backend
    volumes:
      - pets-data:/var/lib/postgresql

networks:
  backend:

volumes:
  pets-data:

Il y a trois sections principales :

  • services. Définit les services de l’application. Ici, deux services sont définis : web et db.

  • networks. Définit les réseaux utilisés par l’application. Ici, un réseau est défini : backend.

  • volumes. Définit les volumes utilisés par l’application. Ici, un volume est défini : pets-data. Le volume est attaché au dossier /var/lib/postgresql (qui se trouve dans le conteneur).

Exercice

Solution

La valeur du champ build est le nom du dossier qui contient le Dockefile. Pour le service web, il s’agit du dossier web ; pour le service db, il s’agit du dossier database. Tous les chemins sont relatifs à la position du fichier docker-compose.yaml lui-même.

Quand on créera l’application en utilisant la configuration spécifiée dans le fichier docker-compose.yaml, on créera deux images, une pour chaque service.

Exercice

Exercise 1.4 Quel sera le nom des deux images ? Quel champ du fichier docker-compose.yaml vous donne cette information ?

Solution

Le nom de l’image correspondant au service web sera pet-store-web.

Le nom de l’image correspondant au service db sera pet-store-db.

Cette information est donnée par le champ image.

2 Pousser une application

Une fois que nous avons créé une application, nous pourrions vouloir la partager en la poussant vers le registre DockerHub ou tout autre registre de conteneurs, qu’il soit privé ou public.

Bon à savoir

Lorsque vous avez installé Docker sur votre ordinateur, vous avez dû créer un compte sur le site DockerHub. Vous aurez besoin de ce compte pour cette activité.

2.1 Renommer les images

Afin de pousser vos images vers le registre, vous devez les renommer, de sorte que le nouveau nom ait la structure suivante :

votre-nom-d'utilisateur/nom-de-l'image:image-tag

Par exemple, puisque mon nom d’utilisateur est quercinigia, je vais renommer les deux images pet-store-web et pet-store-db via la commande docker tag comme suit :

docker tag pet-store-web quercinigia/pet-store-web:1.0

docker tag pet-store-db quercinigia/pet-store-db:1.0

J’ai choisi 1.0 comme tag, mais n’hésitez pas à en choisir un autre.

2.2 Se connecter

Vous devez vous connecter à votre compte DockerHub.

Instructions pour Docker Desktop (macOS et Windows)

  • Utilisateurs mcOS: Regardez cette vidéo.

  • Utilisateurs Windows: Regardez cette vidéo.

Instructions pour la machine virtuelle Linux

Connectez-vous au DockerHub via la commande suivante (remplacez YOUR-USERNAME avec votre nom d’utilisateur):

docker login -u YOUR-USERNAME

Attention

Il se peut que vous receviez un avertissement indiquant que votre mot de passe sera stocké en clair. Il existe des méthodes pour éviter cela. Ces méthodes sortent du cadre de ce cours, vous pouvez lire davantage ici.

2.3 Pousser les images

Une fois que vous avez réussi à vous connecter, vous pouvez saisir les commandes suivantes dans le terminal pour pousser les deux images de votre application :

docker push VOTRE-USERNAME/pet-store-web:1.0

docker push VOTRE-USERNAME/pet-store-db:1.0

N’oubliez pas de remplacer VOTRE-USERNAME par votre nom d’utilisateur.

Une fois la tâche terminée, vérifiez que les images apparaissent dans le registre Docker.

Bon à savoir

Ici, les deux images ont été étiquetées et poussées manuellement. Cette approche devient rapidement fastidieuse dès lors que l’application comporte plus de deux images. Une autre façon de procéder est la suivante :

  • Spécifiez les noms des images avec votre nom d’utilisateur dans le fichier docker-compose.yaml. Par exemple, dans mon docker-compose.yaml j’écrirais :
services:
  web:
    build: web
    image: quercinigia/pet-store-web:1.0
  ...
  db:
    build: database
    image: quercinigia/pet-store-db:1.0
  ...
  • Créez l’application via la commande suivante :

docker compose build

  • Poussez les images de l’application via la commande suivante :

docker compose push

De cette manière, nous pouvons pousser autant d’images que nous souhaitons via une seule commande.

3 Introduction à Kubernetes

Kubernetes est aujourd’hui l’orchestrateur de conteneurs le plus largement utilisé. Il est utilisé pour gérer une application multiservice, généralement déployée sur plusieurs hôtes composant un cluster.

En utilisant Docker Compose, un service correspond à une image Docker et une instance de service à un conteneur Docker.

En revanche, dans Kubernetes, l’unité de calcul est un Pod, c’est-à-dire une collection de conteneurs. En d’autres termes, nous ne raisonnons plus en termes de conteneurs, mais en termes de Pods. Bien entendu, un Pod peut également être constitué d’un seul conteneur.

Bon à savoir

En pratique, nous avons rarement à manipuler des Pods directement dans Kubernetes, car il existe des objets plus avancés qui les gèrent. Nous utiliserons ces objets dans les sections suivantes.

3.2 Déploiement d’un Pod

Pour déployer un Pod, il faut d’abord lui donner sa spécification, c’est-à-dire son nom et les conteneurs qui le composent avec leurs paramètres. De manière similaire à Docker Compose, un Pod est spécifié de manière déclarative avec un fichier de configuration YAML.

Considérons la spécification suivante.

apiVersion: v1
kind: Pod
metadata:
  name: web-pod
  labels:
    app: web-pod
spec:
  containers:
  - name: web
    image: nginx:alpine
    ports:
    - containerPort: 80

Voici une explication des champs de la spécification.

  • apiVersion. Définit la version du schéma de cette spécification.

  • kind. Le type de ressource que nous avons l’intention de créer.

  • metadata. Les métadonnées de la ressource. La liste de toutes les métadonnées est spécifiée ici.

  • spec. La spécification du comportement souhaité du Pod. La liste des spécifications possibles se trouve ici.

Essentiellement, la spécification précédente définit un Pod avec un conteneur qui est lancé à partir de l’image nginx:alpine (un serveur Web) et qui écoute le port 80.

Activité

  • Copiez la spécification ci-dessus dans un fichier nommé sample-pod.yaml (ou tout autre nom de votre choix).

  • En utilisant la commande cd dans le terminal, placez-vous dans le répertoire où se trouve le fichier sample-pod.yaml.

  • Déployez le Pod en saisissant la commande suivante :

kubectl apply -f sample-pod.yaml

  • Vérifiez que le Pod est déployé en saisissant la commande suivante :

kubectl get pods

Lors de la première exécution de cette commande, le Pod peut ne pas encore être prêt. Il est nécessaire d’attendre le téléchargement de l’image nginx:alpine depuis Docker Hub. Patientez quelques secondes, puis relancez la commande jusqu’à ce que le Pod soit indiqué comme étant en cours d’exécution (état Running).

Vous pouvez également obtenir plus d’informations sur le Pod en cours d’exécution (par exemple, l’adresse IP qui lui a été attribuée) en saisissant la commande suivante :

kubectl get pod -o wide web-pod

Exercice

Exercise 3.1 Saisissez l’URL http://localhost:80.

  • Qu’obtenez-vous ? Que se passe-t-il si vous essayez d’utiliser l’IP du Pod au lieu de locahost ?

  • Que devrions-nous définir pour résoudre le problème ?

Solution

  • La connexion est refusée à la fois lorsque l’on utilise localhost et l’adresse IP du Pod.

  • Afin de se connecter au Pod, nous devons définir un objet Service dans Kubernetes, auquel Kubernetes associe une adresse IP que les applications clientes peuvent utiliser.

La configuration suivante définit un service de type LoadBalancer:

apiVersion: v1
kind: Service
metadata:
  name: nginx-service
spec:
  type: LoadBalancer
  ports:
  - port: 8080
    targetPort: 80
    protocol: TCP
    name: http
  selector:
      app: web-pod

Bon à savoir

Un service LoadBalancer est un service NodePort qui offre une fonctionnalité d’équilibrage de charge. Il est associé à une adresse IP que les applications clientes (externes au cluster Kubernetes) peuvent utiliser pour accéder au service.

Exercice

Exercise 3.2 Qu’est-ce le champ selector dans la définition du Service ?

Solution

Le selector est utilisé en interne par Kubernetes pour identifier les Pods qui sont les instances du service. Si vous regardez la spécification du Pod ci-dessus, le champ metadata contient un champ labels. L’une des étiquettes est app : web-pod ; le selector correspond à cette étiquette.

Exercice

Exercise 3.3 Quelle est la signification de port et targetPort ?

Solution

  • port spécifie le numéro de port où le Service sera disponible (celui utilisé par les clients externes).

  • targetPort spécifie le numéro de port ouvert sur l’interface réseau du Pod.

Activité

  • Copiez et collez la spécification du service dans un fichier nommé sample-service.yaml (ou tout autre nom de votre choix).

  • En utilisant la commande cd dans le terminal, placez-vous dans le répertoire où se trouve le fichier sample-service.yaml.

  • Déployez le service via la commande suivante :

kubectl apply -f sample-service.yaml

  • Vérifiez que le service est déployé via la commande suivante :

kubectl get services

Cette commande renvoie tous les services en cours d’exécution dans Kubernetes. Pour chaque service, vous obtenez également un nom. Pour cibler uniquement le service que vous venez de créer, saisissez simplement la commande suivante :

kubectl get svc/nginx-service

Le nom de votre service est nginx-service (comme spécifié dans le fichier sample-service.yaml) ; `svc`` est seulement l’espace de noms où tous les services Kubernetes sont placés.

Examinez maintenant la sortie de la commande kubectl get svc/nginx-service.

Exercice

Exercise 3.4 Pourquoi le service est-il associé à deux adresses IP (CLUSTER-IP et EXTERNAL-IP)?

Solution

Lorsqu’on demande à Kubernetes de créer un service de type LoadBalancer, un service interne de type ClusterIP est également mis en place automatiquement. Les pods qui assurent le traitement des requêtes sont exposés derrière ce service ClusterIP. Ainsi, la valeur CLUSTER-IP correspond à l’adresse IP du service interne au cluster, tandis que la valeur EXTERNAL-IP désigne l’adresse IP du LoadBalancer, utilisée par les applications externes pour accéder aux fonctionnalités fournies par les pods. Par conséquent, en saisissant dans un navigateur cette adresse EXTERNAL-IP, suivie du port 8080, on doit pouvoir afficher la page d’accueil de Nginx.

Exercice

Exercise 3.5 La colonne PORT(S) devrait indiquer une valeur du type 8080:31355 (le port 31355 peut être différent pour vous). Pouvez-vous expliquer à quoi correspond ce port ?

Solution

Lorsque nous demandons la création d’un service de type LoadBalancer, Kubernetes crée aussi un service de type NodePort. Le port 31355 est ouvert sur chaque nœud du cluster Kubernetes (d’où le nom de NodePort).

En pratique, lorsque nous saisissons http://localhost:8080, l’équilibreur de charge le remplace par l’adresse IP d’un nœud du cluster Kubernetes, suivie du numéro de port 31355. Lorsque le kube-proxy de ce nœud reçoit cette demande, il la transmet au service ClusterIP approprié.

Ce mécanisme est illustré à la diapositive 80 du cours 3.

Exercice

Exercise 3.6 Supprimez le Pod avec la commande suivante :

Ensuite saisissez la commande suivante :

Le Pod est-il toujours en cours d’exécution ? Sachant que l’un des rôles d’un orchestrateur est de maintenir l’état souhaité de l’application, ce que vous observez vous surprend-il ?

Solution

Même si l’orchestrateur a bien pour mission de maintenir l’état souhaité de l’application, le Pod demeure un objet relativement élémentaire, qui ne dispose pas nativement d’un mécanisme de redémarrage en cas d’arrêt. Pour bénéficier d’un véritable mécanisme d’auto-réparation (self-healing), il est nécessaire d’utiliser des objets de type ReplicaSet ou Deployment.

4 Kubernetes : déploiement d’une application

Dans cette section, nous allons déployer notre animalerie en ligne sur Kubernetes. Pour rappel, notre application se compose de deux services : web et db.

Pour définir une application dans Kubernetes, nous devons utiliser deux types d’objets pour chaque service de l’application :

  • Une ressource de charge de travail qui donne la spécification du service, telle que les Pods qui composent le service lui-même (images, paramètres réseau) et les métadonnées, telles que le nombre d’instances souhaité.

  • Un objet Service. Comme nous l’avons vu précédemment, un objet Service est associée à une adresse IP qui permet aux applications clientes, à l’intérieur et à l’extérieur du cluster Kubernetes, de se connecter au service.

Lorsque nous définissons une application dans Kubernetes, nous avons rarement, voire jamais, besoin de manipuler directement des Pods. Nous pouvons faire recours à des objets plus avancés, appelés Contrôleurs, pour une définition de l’état souhaité de l’application elle-même. Le type de contrôleur que nous devons utiliser dépend de la nature du service lui-même : sans état ou avec état.

Exercice

Exercise 4.1

  • Le service web de notre application est-il avec ou sans état ?

  • Le service db de notre application est-il sans état ou avec état ?

Solution

Le service web est sans état.

Le service db est avec état.

4.1 Le service web : Deployment

Pour les services sans état, nous pouvons utiliser un objet Deployment pour le déploiement d’un groupe de Pods identiques qui seront lancés sur plusieurs nœuds du cluster Kubernetes.

Nous spécifions un Deployment correspondant au service web.

apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
spec:
  replicas: 3
  selector:
    matchLabels:
      app: pets
      service: web
  template:
    metadata:
      labels:
        app: pets
        service: web
    spec:
      containers:
      - image: quercinigia/pet-store-web:1.0
        name: web
        ports:
        - containerPort: 3000
          protocol: TCP

Voici l’explication de la spécification :

  • Un Deployment nommé web est créé, comme indiqué dans le champ metadata.name.

  • Le Deployment crée trois Pods identiques, comme indiqué dans le champ spec.replicas.

  • Le Deployment gère tous les Pods ayant les deux labels app : pets et service : web. Ceci est indiqué dans le champ spec.selector.matchLabels. Ce champ est utilisé pour identifier les Pods faisant partie de ce Deployment.

  • La configuration des Pods qui font partie de ce Deployment est donnée dans le champ spec.template et ses sous-champs.

  • Chaque Pod du Deployment a les labels app : pets et service : web. Ceci est indiqué dans le champ spec.template.metadata.labels. Notez que nous spécifions ici exactement les mêmes valeurs que nous avons indiquées dans le champ spec.selector.matchLabels.

  • Chaque Pod a exactement un conteneur, nommé web, exécuté à partir de l’image quercinigia/pet-store-web:1.0 stockée sur le DockerHub. Le conteneur écoutera le port 3000 et utilisera le protocole TCP. Ceci est spécifié dans le champ spec.template.spec.containers.

Activité

  • Copiez et collez la spécification précédente dans un nouveau fichier web-deployment.yaml (ou tout autre nom de votre choix). Assurez-vous de remplacer l’image quercinigia/pet-store-web:1.0 avec celle que vous avez poussée vers votre registre DockerHub.

  • Utilisez la commande cd dans le terminal pour vous positionner dans le répertoire où le fichier web-deployment.yaml est stocké.

  • Déployez ce Deployment dans Kubernetes en saisissant la commande suivante :

kubectl apply -f web-deployment.yaml

Nous pouvons clairement voir trois types d’objets :

4.2 Le service web : Service

Nous allons maintenant définir un Service pour que les utilisateurs puissent accèder au service Web. Voici la définition :

apiVersion: v1
kind: Service
metadata:
  name: web
spec:
  type: LoadBalancer
  ports:
  - port: 8080
    targetPort: 3000
    protocol: TCP
  selector:
    app: pets
    service: web

Exercice

Exercise 4.4 Décrivez la spécification de ce service.

Solution

  • Nous créons un service nommé web, comme l’indique le champ metadata.name.

  • Le service est de type LoadBalancer, conformément au champ spec.type.

  • Le service écoute le port 8080 et utilise le protocole TCP. Chaque instance de service (c.-à-d. chaque Pod) écoute le port 3000. Ces informations sont précisées dans le champ spec.ports.

  • Les instances du service (autrement dit, les Pods) sont celles qui portent des étiquettes app: pets et service: web. Cela est défini dans le champ spec.selector.

Activité

  • Créez le service avec la commande suivante :

kubectl apply -f web-service.yaml

4.3 Le service db : StatefulSet

Kubernetes a défini un type spécial de ReplicaSet pour les services avec état, appelé StatefulSet.

Mous allons maintenant définir un StatefulSet pour donner la spécification du service db.

apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: db
spec:
  selector:
    matchLabels:
      app: pets
      service: db
  serviceName: db
  template:
    metadata:
      labels:
        app: pets
        service: db
    spec:
      containers:
      - image: quercinigia/pet-store-db:1.0
        name: db
        ports:
        - containerPort: 5432
        volumeMounts:
        - mountPath: /var/lib/postgresql
          name: pets-data
  volumeClaimTemplates:
  - metadata:
      name: pets-data
    spec:
      accessModes:
      - ReadWriteOnce
      resources:
        requests:
          storage: 100Mi

La signification des champs de cette spécification devrait vous être familière. Ils sont également documentés dans la documentation officielle de Kubernetes. Deux points méritent toutefois d’être précisés.

Premièrement, la spécification d’un StatefulSet requiert l’attribut serviceName, qui désigne le service responsable de l’identité réseau des Pods gérés par le StatefulSet. Cet attribut est obligatoire.

Deuxièmement, le champ volumeClaimTemplates permet de définir les besoins en stockage d’un Pod. La demande entraîne la création d’un volume pets-data (où PostgreSQL conserve ses données). Les paramètres suivants sont définis :

  • Le mode d’accès ReadWriteOnce signifie que le volume peut être monté en lecture-écriture par un seul nœud du cluster Kubernetes. Les modes d’accès sont documentés ici.

  • Nous demandons au moins 100 Mo de stockage pour la base de données.

Activité

  • Copiez et collez la spécification précédente dans un nouveau fichier db-stateful-set.yaml (ou un nom de votre choix). Assurez-vous de remplacer l’image quercinigia/pet-store-db:1.0 avec celle que vous avez poussée vers votre registre DockerHub.

  • Utilisez la commande cd dans le terminal pour vous positionner dans le répertoire où le fichier db-stateful-set.yaml est stocké.

  • Déployez ce StatefulSet dans Kubernetes en saisissant la commande suivante :

kubectl apply -f db-stateful-set.yaml

5.2 Conclusion

Dans les exercices précédents, nous avons déployé une application à deux services, ce qui nous a amenés à créer quatre fichiers et à exécuter autant de commandes. Pour des applications plus grandes, cette approche devient vite fastidieuse. Une alternative consiste à regrouper toutes les définitions dans un seul fichier (par exemple, pets.yaml), où chaque spécification est séparée par ---.

Voici un exemple dans lequel nous écrivons les spécifications du Service et du Deployment associés au service web.

apiVersion: v1
kind: Service
metadata:
  name: web
spec:
  type: LoadBalancer
  ports:
  - port: 8080
    targetPort: 3000
    protocol: TCP
  selector:
    app: pets
    service: web
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
spec:
  replicas: 3
  selector:
    matchLabels:
      app: pets
      service: web
  template:
    metadata:
      labels:
        app: pets
        service: web
    spec:
      containers:
      - image: quercinigia/pet-store-web:1.0
        name: web
        ports:
        - containerPort: 3000
          protocol: TCP
