Modélisation multidimensionnelle

L’objectif de ce tutoriel est d’apprendre les stratégies de modélisation multidimensionnelle pour un data warehouse.

Scénario applicatif

Nous voulons mettre en place un data warehouse pour une chaîne de magasins de location de DVD, que nous appellerons Sakila. Chaque magasin de Sakila a sa propre base de données opérationnelle, où les informations sur les locations sont stockées. La direction de Sakila souhaite agréger les données de tous les magasins dans un entrepôt de données centralisé (un data warehouse), afin de pouvoir analyser ces données et vérifier les tendances commerciales.

Le modèle physique de la base de données opérationnelle de Sakila est illustré en figure 1.

Figure 1: Modèle physique de la base de données Sakila

Méthodologie de modélisation

Ralph Kimball, dans son ouvrage The Data Warehouse Toolkit: The definitive guide to dimensional modeling, propose une méthodologie en quatre étapes pour modéliser un data warehouse.

1. Identifier les processus métier.

Un processus métier est une activité de bas niveau effectuée par une organisation, telle que la prise de commandes, la facturation, la réception des paiements, l’inscription des étudiants, la réalisation d’une procédure médicale ou le traitement des réclamations. Un processus métier est normalement associé à un système opérationnel, comme par exemple un système de gestion des commandes ou un système de facturation.

Il faut aussi souligner qu’un processus métier ne correspond pas à un département de l’organisation, ni à l’une de ses fonctions.

2. Déclarer le grain

Déclarer le grain signifie spécifier exactement ce que représente une ligne individuelle de la table des faits. Le grain indique le niveau de détail associé aux mesures de la table des faits. Il répond à la question : « Comment décrivez-vous une seule ligne dans la table des faits ? » Le grain est déterminé par les réalités physiques du système opérationnel qui capture les événements du processus métier. Exemples de déclaration du grain : une ligne pour chaque article sur une facture d’un médecin, une ligne pour chaque carte d’embarquement scannée à la porte d’embarquement, une ligne pour chaque transaction bancaire effectuée par un client….

Kimball recommande de choisir le grain le plus détaillé possible, car cela maximise la flexibilité analytique du data warehouse.

3. Identifier les dimensions

Si le grain est bien défini, les dimensions peuvent généralement être facilement identifiées, car elles représentent le « qui, quoi, où, quand, pourquoi et comment » associés à un événement métier. Des exemples de dimensions courantes incluent : date, produit, client, employé… Pour chaque dimension choisie, il faudra ensuite lister tous les attributs discrets, de type texte, qui décrivent chaque table de dimension.

4. Identifier les faits

Les faits sont déterminés en répondant à la question : « Qu’est-ce que le processus mesure ? » Les utilisateurs métier sont particulièrement intéressés par l’analyse de ces indicateurs de performance. Tous les faits candidats dans une conception doivent être cohérents avec le grain défini à l’étape 2. Les faits qui ont un grain différent doivent être placés dans une table de faits séparée. Normalement les faits sont des valeurs numériques additives, telles que la quantité commandée ou le montant en dollars.

Il est important de déterminer les faits non seulement à partir de l’analyse des données opérationnelles existantes, mais aussi après une interaction approfondie avec les utilisateurs métier.

Identification du processus métier

Exercice

Exercise 1 Quel est le processus métier que nous voulons modéliser dans le data warehouse de Sakila?

Solution

Le processus métier que nous voulons modéliser est la location de films. Un autre processus métier pourrait être l’approvisionnement des magasins en films, mais nous ne le considérons pas ici.

Déclaration du grain

Exercice

Exercise 2 Déclarez le grain du data warehouse de Sakila.

Solution

Chaque fois qu’un client loue un film dans un magasin, un enregistrement est créé dans la base de données opérationnelle. Donc, le grain est : une ligne pour chaque location de film par un client dans un magasin à un moment donné.

Identification des dimensions

Exercice
Exercise 3 Identifiez les dimensions du data warehouse de Sakila.
Solution

Les dimensions sont :

  • La dimension client : qui loue le film?
  • La dimension film : quel film est loué?
  • La dimension période : quand le film est-il loué?
  • La dimension magasin : dans quel magasin (où) le film est-il loué?
  • La dimension employé : quel employé (qui) a géré la location?

Identification des faits

Exercice

Exercise 4 Identifiez les faits du data warehouse de Sakila.
Solution

Concernant une location de film par un client dans un magasin à un moment donné, nous souhaitons connaître le montant payé pour la location et la durée de la location.

Modélisation dimensionnelle

Maintenant que nous avons détermine le grain, identifié les faits et les dimensions, nous dessinons progressivement le schéma en étoile pour le data warehouse de Sakila.

Exercice

Exercise 5 Déssinez le premier croquis du schéma en étoile pour le data warehouse de Sakila.

Solution

Exercice

Exercise 6 Identifiez la clé primaire de la table des faits.

Solution

La clé primaire de la table des faits est composée de tous les attributs qui réfèrent aux dimensions. Donc, nous écrivons un attribut pour chaque dimension :

  • staff_pk, clé étrangère reférençant la dimension dim_staff.

  • film_pk, clé étrangère reférençant la dimension dim_film.

  • date_pk, clé étrangère reférençant la dimension dim_date.

  • store_pk, clé étrangère reférençant la dimension dim_store.

  • customer_pk, clé étrangère reférençant la dimension dim_customer.

Voici le nouveau schéma.

Exercice

Exercise 7 Pour chaque table de dimension, il y a une table correspondante dans la base de données opérationnelle.

Il faudra décider quelles colonnes ajouter dans chaque table de dimension. En particulier, il faudra discuter les points suivants :

  1. Est-ce qu’on va ajouter dans une table de dimension la clé primaire de la table opérationnelle correspondante ?

  2. Comment va-t-on traiter les colonnes qui sont des clés étrangères dans la table opérationnelle correspondante ?

Solution

  • Concernant la première question, la réponse est oui. Chaque table de dimension doit contenir la clé primaire de la table opérationnelle correspondante. Cela permet de faire le lien entre le data warehouse et la base de données opérationnelle.

  • Concernant la deuxième question, il y a deux possibilités.

  1. Soit on décide d’intégrer les dimensions liées, en ajoutant les attributs des tables liées dans la table de dimension courante.
  2. Soit on décide de ne pas intégrer les dimensions liées, et dans ce cas on ajoute dans la table de dimension courante les clés étrangères qui réfèrent aux dimensions liées.

Si on choisit la deuxième option, on obtient un schéma en flocon (snowflake schema), qui est normalisé.

Pour un data warehouse, on préfère généralement la première option (donc un schéma dénormalisé en étoile), car elle permet d’éviter les jointures entre les tables de dimensions.

Une dernière remarque sur l’attribut film_categories dans la table dim_film. La valeur de cet attribut est une liste. On pourrait vouloir éviter cela en utilisant un one-hot encoding. En d’autres termes, si l’ensemble de toutes les catégories possibles est petit, on peut avoir un attribut booléen pour chaque catégorie ; une valeur True indiquerait que le film appartient à la catégorie correspondante.

Exercice

Exercise 8 Identifiez et ajoutez les faits dans la table des faits.

Solution

La dimension des dates peut être enrichie avec des attributs supplémentaires, tels que le jour de la semaine, le mois, le trimestre, l’année fiscale, un indicateur de jour férié, etc.
