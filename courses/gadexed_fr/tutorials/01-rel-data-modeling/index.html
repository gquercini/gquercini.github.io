Gestion et analyse des données — Tutoriel 1

Modélisation des données relationnelles

L’objectif de ce tutoriel est de s’exercer à la conception d’une base de données relationnelle.

Les sujets suivants sont abordés :

  • Schéma conceptuel d’une base de données relationnelle.

  • Schéma logique d’une base de données relationnelle.

  • Clés primaires et clés étrangères.

Scénario applicatif

Nous souhaitons gérer les données d’un système de réservation de voyages avec des clients partout dans le monde. Lors de leur inscription, les clients se voient automatiquement attribuer un identifiant numérique et ils doivent indiquer leur prénom et nom de famille, leur genre, date de naissance, un numéro de téléphone, une adresse e-mail et leur pays de résidence.

Tout client peut réserver un voyage qui inclut la réservation de un ou plusieurs vols et, éventuellement, un ou plusieurs hôtels

Exemple

Alice souhaite voyager de Paris, France à New York City (NYC), USA et elle prévoit de rester à NYC pendant 10 jours. Son voyage inclut deux vols : un vol aller de Paris à NYC et un vol retour de NYC à Paris ; ainsi qu’un hôtel à NYC

Un vol est opéré par une compagnie aérienne, dont le système conserve le nom (par exemple, British Airways), le pays où la compagnie est incorporée et, lorsque disponible, son code IATA (par exemple, BA, un code à deux lettres identifiant la compagnie), son code ICAO (par exemple, BAW, un code à trois lettres identifiant la compagnie) et un nom alternatif ou alias (par exemple, British).

Un vol relie deux aéroports, chacun ayant un nom (par exemple, London Heathrow Airport), et, éventuellement, un code IATA (par exemple, LHR) et un code ICAO (par exemple, EGLL) ; un aéroport dessert un lieu spécifique (par exemple, London, UK) et sa position précise est donnée par ses coordonnées géographiques (latitude et longitude).

Un vol reliant deux aéroports à des heures de départ et d’arrivée spécifiques est identifié par un numéro de vol.  Deux vols opérés par deux compagnies aériennes différentes ne peuvent pas avoir le même numéro de vol, mais le même numéro de vol peut désigner deux vols opérés par la même compagnie aérienne à des jours différents.

Exemple

Le vol Emirates EK074 part de Paris, France à 10h00 et arrive à Dubaï, Émirats arabes unis à 19h40 (indépendamment du jour de départ).

Pour chaque vol réservé par un client, le système conserve le numéro de siège, la classe de voyage (par exemple, économique ou affaires), le prix et la date du vol.  Habituellement, les compagnies aériennes incluent des détails sur le type d’avion qu’elles prévoient d’utiliser dans leurs horaires de vol ; ces détails incluent le nom de l’avion (par exemple, Boeing 787-8) et, lorsque disponible, le code IATA (par exemple, 788, un identifiant unique à trois lettres pour l’avion) et le code ICAO (par exemple, B788, un identifiant unique à quatre lettres pour l’avion).

Le système conserve une liste d’hôtels, avec leurs noms et adresses. Les clients peuvent écrire un avis pour un hôtel ; dans ce cas, le système stocke le texte de l’avis, la date et son auteur. Lorsque un client réserve un hôtel, le système conserve le prix payé, les dates d’arrivée et de départ et si le petit-déjeuner est inclus.

Schéma conceptuel

Le schéma conceptuel de la base de données est une représentation abstraite des entités de l’application et de leurs associations. Le schéma conceptuel ne comprend aucun détail de mise en œuvre, comme la définition des clés primaires ou des clés étrangères, ni aucune information sur les types des attributs. Le schéma conceptuel est généralement représenté à travers un diagramme Entité-Association (ang., Entity-Relationship, ER).

Exercice

Exercise 1 Proposez un diagramme Entité-Association (ER) décrivant le modèle conceptuel d’une base de données relationnelle pour le contexte applicatif donné.

  • Identifiez les entités et les associations pertinentes.

  • Identifiez les attributs pour chaque entité et chaque association.

  • Pour chaque entité, soulignez les attributs composant la clé primaire.

  • Déterminez les cardinalités minimales et maximales pour chaque association.

Schéma logique

Le schéma logique d’une base de données relationnelle est l’ensemble des tables qui implémentent le schéma conceptuel. Le schéma logique inclut la définition des clés primaires et des clés étrangères, mais il n’inclut aucune information sur les types des colonnes.

Exercice

Exercise 2 Donnez le schéma logique de la base de données. Pour chaque table, soulignez la clé primaire et spécifiez les clés étrangères.

Solution

customer (cust_code, first_name, last_name, gender, birth_date, phone_number, email_address, cust_country)

trip (trip_code, cust_code)

flight (flight_id, flight_number, flight_date, departure_time, arrival_time, departure_airport, arrival_airport, airline_name, aircraft_name)

hotel (hotel_id, hotel_name, hotel_address)

hotel_reservation (trip_code, hotel_id, price, checkin_date, checkout_date, breakfast)

flight_booking (trip_code, flight_id, seat_number , travel_class, booking_date, price)

review (hotel_id, cust_code, content, date)

airline (airline_name, airline_country, airline_iata, airline_icao, airline_alias)

aircraft (aircraft_name, aircraft_iata, aircraft_icao)

airport (airport_id, airport_name, airport_iata, airport_icao, city, country, latitude, longitude)

Clés étrangères :

trip(cust_code) → customer(cust_code)

flight(departure_airport) → airport(airport_id)

flight(arrival_airport) → airport(airport_id)

flight(airline_name) → airline(airline_name)

flight(aircraft_name) → aircraft(aircraft_name)

hotel_reservation(hotel_id) → hotel(hotel_id)

flight_booking(trip_code) → trip(trip_code)

flight_booking(flight_id) → flight(flight_id)

review(cust_code) → customer(cust_code)

review(hotel_id) → hotel(hotel_id)

Dans la table flight, nous avons introduit une clé primaire substitutive flight_id pour identifier de manière unique chaque instance de vol. Cependant, la table n’est pas en troisième forme normale (3NF), car les attributs departure_airport et arrival_airport dépendent de flight_number.
