Cassandra

L’objectif de ce tutoriel est de vous familiariser avec Apache Cassandra et son langage de requête CQL (Cassandra Query Language).

Configuration de l’environnement

  • Cliquez sur l’icône de Docker Desktop que vous trouvé sur le bureau.

  • Ouvrez GitBas et saisissez la commande suivante pour démarrer un conteneur Docker avec Cassandra:

    docker compose -f docker-compose-cassandra.yml up -d

    Cela va démarrer un conteneur Docker en arrière-plan avec le serveur Cassandra.

    • Attendez quelques instants que le conteneur démarre complètement.

  • Pour accéder au shell CQL de Cassandra à l’intérieur du conteneur, utilisez la commande suivante:

  docker exec -it cassandra-container cqlsh

Lorsque vous avez démarré le shell, vous n’avez pas spécifié le noeud Cassandra auquel vous voulez vous connecter. Par défaut, il se connecte au noeud local que vous avez démarré dans le conteneur Docker. Le shell indique aussi que vous êtes connecté au cluster “Test Cluster” ; c’est le nom d’un cluster par défaut consistant en un seul noeud.

Pour visualiser la liste des commandes disponibles dans le shell CQL, vous pouvez taper HELP; et appuyer sur Entrée.

Voici une liste de commandes utiles pour interagir avec Cassandra.

Décrire l’environnement

  1. Pour afficher les informations sur le cluster auquel vous êtes connecté, utilisez la commande suivante:
DESCRIBE CLUSTER;

Les informations affichées incluent :

  • Le nom du cluster.
  • L’algorithme de partitionnement utilisé.
  • Le type de snitch utilisé. Le snitch est un composant qui détermine comment les noeuds sont trouvés et comment router les requêtes dans le cluster.
  1. Pour afficher la liste des keyspaces (espaces de clés) disponibles dans le cluster, utilisez la commande suivante :
DESCRIBE KEYSPACES;

Vous devriez trouver une liste de de keyspaces de sytème, que Cassandra gère internement et qui ne sont pas destinés à être modifiés par les utilisateurs. Cassandra utilise ces keyspaces pour stocker des informations sur le cluster, les métadonnées, les informations de sécurité, etc.

Création d’un keyspace

Les commandes CQL peuvent parfois être un peu verbeuses, mais Cassandra offre une fonctionnalité de auto-complétion pour faciliter leur saisie.

Saisissez la commande suivante suivie par un caractère de tabulation :


CREATE KEYSPACE my_keyspace WITH

Vous devriez voir que la commande est complétée automatiquement de la manière suivante :

CREATE KEYSPACE my_keyspace WITH replication = {'class': '

Si vous appuyez à nouveau sur la touche de tabulation, vous verrez une liste des options disponibles pour la stratégie de réplication.

Choisissez la stratégie SimpleStrategy.

Ensuite, saisissez le caractère de tabulation et spécifiez un facteur de réplication de 1 (car nous avons un seul noeud dans notre cluster).

Ajouter une accolade et un point-virgule pour terminer la commande et exécutez-la.

Maintenant, saisissez la commande suivante pour vérifier que le keyspace a été créé avec succès:

DESCRIBE KEYSPACE my_keyspace;

Vous devriez voir la définition du keyspace que vous venez de créer, y compris la stratégie de réplication et le facteur de réplication. Une troisième option s’affiche égalemenet, durable_writes, qui est activée par défaut.

Maintenant, utilisez la commande suivante pour sélectionner le keyspace que vous venez de créer:

USE my_keyspace;

Création d’une table

Nous allons ensuite créer une table dans ce keyspace. Saisissez la commande suivante :

CREATE TABLE user ( first_name text, last_name text, 
    middle_initial text, title text, 
    PRIMARY KEY (last_name, first_name)) ;

La commande précédente crée une table nommée user avec quatre colonnes : first_name, last_name, middle_initial et title. La clé primaire est composée de last_name et first_name, où last_name est la clé de partition et first_name est la clé de clustering.

Pour vérifier que la table a été créée avec succès, utilisez la commande suivante:

DESCRIBE TABLE user;

Insérer des données

Pour insérer des données dans une table, il faut utiliser la commande INSERT INTO.

INSERT INTO user (first_name, last_name, middle_initial, title) VALUES ('John', 'Doe', 'M.', 'Data Engineer');

Nous pouvons vérifier que les données ont été insérées avec succès en utilisant la commande SELECT:

SELECT * FROM user ;

Exercice

Exercise 1 Essayez d’insérer un autre utilisateur dans la table user sans spécifier la valeur pour la colonne middle_initial. Est-ce que l’insertion réussit ? Pourquoi ?

Solution

Oui, l’insertion réussit car il est possible de ne pas spécifier de valeur pour une colonne qui ne fait pas partie de la clé primaire.

INSERT INTO user (first_name, last_name, title) VALUES ('Jane', 'Smith', 'Data Scientist');

Exercice

Exercise 2 Essayez d’insérer un autre utilisateur dans la table user sans spécifier la valeur pour la colonne first_name. Est-ce que l’insertion réussit ? Pourquoi ?

Solution

Non, l’insertion échoue car first_name fait partie de la clé primaire et doit toujours avoir une valeur.

INSERT INTO user (last_name, title) VALUES ('Jane', 'Data Scientist');

Lorsqu’on insère des données dans une table Cassandra, il est obligatoire de fournir des valeurs pour toutes les colonnes qui font partie de la clé primaire.

Exercice

Exercise 3 Essayez d’insérer un autre utilisateur ayant le même last_name que l’utilisateur existant, mais un first_name différent. Est-ce que l’insertion réussit ? Pourquoi ?

Solution

Oui, l’insertion réussit car la combinaison de last_name et first_name est unique.

INSERT INTO user (last_name, first_name, middle_initial, title) VALUES ('Doe', 'Peter', 'Cloud architect', 'Dr.');

Exercice

Exercise 4 Essayez d’insérer un autre utilisateur ayant le même last_name et `first_name`` que un utilisateur existant. Est-ce que l’insertion réussit ? Pourquoi ?

Solution

Oui, l’insertion réussit, mais elle écrase les données de l’utilisateur existant.

INSERT INTO user (last_name, first_name, middle_initial, title) VALUES ('Doe', 'Joe', 'Mr.', 'Developer.');

Cette opération est appelée upsert (update + insert) dans Cassandra.

Mettre à jour des données

Pour mettre à jour des données dans une table, on utilise la commande UPDATE.

UPDATE user SET title = 'Senior Data Engineer' WHERE last_name = 'Doe' and first_name = 'John' ;

Exercice

Exercise 5 Essayez de changer le titre de l’utilisateur avec le last_name ‘Doe’ sans spécifier le first_name. Est-ce que la mise à jour réussit ? Pourquoi ?

Solution

Non, la mise à jour échoue car il est obligatoire de fournir des valeurs pour toutes les composantes de la clé primaire dans la clause WHERE.

UPDATE user SET title='unknown' where last_name='Doe';

On a vu précédemment qu’une insertion se comporte comme une opération de mise à jour si on insère une ligne avec une clé primaire existante. Puisqu’il n’y a pas de différence entre une insertion et une mise à jour dans Cassandra, il est logique que les mêmes règles s’appliquent aux deux opérations.

Colonnes

Une colonne est l’unité de base de stockage des données dans Cassandra. Une colonne est identifiée par un nom et a une valeur associée. Les valeurs d’une colonne doivent être du même type de données pour toutes les lignes d’une table. Deux attributs importants d’une colonne sont les timestamps et les TTL (Time To Live).

Timestamps

A chaque fois qu’une colonne est insérée ou mise à jour, Cassandra associe un timestamp (en microsecondes) à cette opération. Internement, Cassandra utilise ces timestamps pour résoudre les conflits lors de la réplication des données entre les noeuds du cluster (last write wins). Pour visualiser les timestamps des colonnes, vous pouvez utiliser la commande SELECT avec la fonction writetime().

SELECT first_name, last_name, middle_initial, writetime(middle_initial) FROM user;

Il n’y a pas de timestamp pour les colonnes qui n’ont pas de valeur (NULL).

Exercice

Exercise 6 Essayez de renvoyer le timestamp pour l’une des colonnes de la clé primaire. Qu’est-ce que vous observez ?

Solution 
SELECT first_name, last_name, writetime(first_name) FROM user;

L’opération n’est pas permise. En effet, les colonnes qui font partie de la clé primaire n’ont pas de timestamp associé, car elles sont immuables. Si vous essayez de modifier le prénom d’un utilisateur avec un UPDATE, vous devez en fait insérer une nouvelle ligne avec une nouvelle valeur pour le prénom.

TTL (Time To Live)

Une des fonctionnalités intéressantes de Cassandra est la possibilité de définir un TTL (Time To Live) pour une colonne. Par défaut, les colonnes n’ont pas de TTL, ce qui signifie que les données sont conservées indéfiniment. La requête suivante renvoie le TTL de la colonne middle_initial pour chaque ligne de la table user:

SELECT ttl(middle_initial) FROM user;

Essayons maintenant de définir un TTL de 60 secondes pour la colonne title de l’utilisateur ‘John Doe’:

UPDATE user USING TTL 60 SET title = 'Temporary Title' WHERE last_name = 'Doe' and first_name = 'John' ;

Après 60 secondes, la colonne title de l’utilisateur ‘John Doe’ sera automatiquement supprimée par Cassandra. Vérifiez-le.

On peut aussi définir un TTL lors de l’insertion d’une nouvelle ligne:

INSERT INTO user (first_name, last_name, middle_initial, title) VALUES ('Alice', 'Johnson', 'A.', 'Intern') USING TTL 60 ;

La commande précédente insère une nouvelle ligne pour l’utilisateur ‘Alice Johnson’ avec un TTL de 60 secondes pour toutes les colonnes. Après 60 secondes, toute la ligne sera automatiquement supprimée par Cassandra.

Lire et écrire des données

Pour insérer des données dans la table user, utilisez la commande suivante:

INSERT INTO user (first_name, last_name, title) VALUES ('John', 'Doe', 'Data Engineer');

Vous pouvez lire les données que vous avez insérées en utilisant la commande suivante :

SELECT * FROM user WHERE last_name = 'Doe' and first_name = 'John' ;

Dans la commande précédente, nous avons spécifié une valeur pour chaque composant de la clé primaire dans la clause WHERE.

Que se passe-t-il si vous essayez d’exécuter la commande suivante ?

SELECT * FROM user WHERE last_name = 'Doe' ;

Tout se passe bien.

Et avec la commande suivante ?

SELECT * FROM user WHERE first_name = 'John' ;

Nous avons une erreur.

Nous pouvons supprimer des données en utilisant la commande DELETE.

DELETE FROM user WHERE last_name = 'Doe' and first_name = 'John' ;

Pour vérifier que les données ont été supprimées, vous pouvez exécuter à nouveau la commande SELECT.

Pour supprimer toutes les lignes d’une table, on peut utiliser la commande TRUNCATE.
Pour supprimer une table, on utilise la commande DROP TABLE.

Utilisation de la clé primaire

Exécutons à nouveau la commande INSERT pour insérer des données dans la table user:

INSERT INTO user (first_name, last_name, title) VALUES ('John', 'Doe', 'Data Engineer');

Lorsque nous avons créé la table user, nous avons défini la clé primaire comme étant composée de last_name et first_name. Cela est interprété par Cassandra comme une clé primaire composite, où last_name est la clé de partition et first_name est la clé de clustering.

Si nous essayons d’insérer une autre ligne avec le même last_name mais un first_name différent, comme dans la commande suivante:

INSERT INTO user (first_name, last_name, title) VALUES ('Jane', 'Doe', 'Data Scientist');

Cassandra va accepter cette insertion car la combinaison de last_name et first_name est unique.

Maintenant, si on cherche toutes les entrées avec last_name égal à ‘Doe’:

SELECT * FROM user WHERE last_name = 'Doe' ;

Nous obtenons un résultat. La requête précédente charge en mémoire le contenu de la partition correspondant à last_name égal à ‘Doe’.

Si on essaie de récupérer les entrées avec first_name égal à ‘John’:

SELECT * FROM user WHERE first_name = 'John' ;

on obtient une erreur. En fait, une recherche basée uniquement sur la clé de clustering n’est pas permise dans Cassandra en raison des performances. Si nous avons vraiment besoin de faire ce type de requête, on peut spécifier la commande suivante :

SELECT * FROM user WHERE first_name = 'John' ALLOW FILTERING;
